# LinkedIn Strategy

## Purpose

This document is **not** my LinkedIn profile. It is the **canonical guide** for how my LinkedIn profile should be written, maintained, and updated.

Future AI assistants should use this file whenever helping improve my LinkedIn headline, About section, experience entries, skills, featured content, or Open To Work settings.

**LinkedIn serves different goals than a résumé:**

| Résumé | LinkedIn |
|--------|----------|
| Tailored to one application | Discovered by many recruiters and hiring managers |
| Compressed to one page | Searchable over time; browsed in sections |
| Achievement bullets | Narrative + credibility + social proof |
| Private until sent | Public professional presence |

**The profile should:**

- Attract recruiters searching for senior engineers with ownership and architecture judgment
- Rank well in LinkedIn search without keyword stuffing
- Tell the **same engineering story** as the Career Knowledge Base—not a conflicting identity
- Demonstrate **ownership, simplification, and maintainability**—not framework hoarding
- Support **Full Stack**, **Senior Frontend**, and **AI Software Engineer** opportunities without narrowing to "React developer only"

**Source of truth for facts:** [`../story-index.md`](../story-index.md) → [`../context.md`](../context.md) → leaf documents (`experience/`, `stories/`, `projects/`). Generate LinkedIn copy from those sources—never invent metrics, dates, or scope.

## Target Audience

### Primary

- **Engineering Managers** — hiring for ownership, collaboration, and long-term code health
- **Senior Recruiters** — filtering on title, skills, and search keywords
- **Startup Founders** — need engineers who ship and improve systems without heavy process
- **Technical Leads** — evaluating architecture judgment and cross-stack collaboration

### Secondary

- **Staff Engineers** — peers, referrals, future collaborators
- **Product Managers** — cross-functional network from past roles
- **Former coworkers** — recommendations, referrals, reconnection
- **Potential collaborators** — open-source, side projects, community

**Write for humans first.** Recruiter search optimization matters, but the profile should still read like a credible engineer—not an SEO page.

## About Section

### What it should communicate

The About section is a **short professional narrative**—conversational, authentic, not marketing copy.

**Lead with themes (in order of emphasis):**

1. **Ownership** — features and systems through production and beyond
2. **Simplifying complex systems** — consolidation, compatibility layers, configurability
3. **Maintainability and architecture** — code that future engineers (and AI tools) can navigate
4. **Product thinking** — product before code; collaboration with PM and Design
5. **Full-stack background** — professional Java/Spring Boot, Node.js, AWS—not hobby backend
6. **AI-assisted engineering** — accelerates work; review and judgment remain human ([`../philosophy/working-with-ai.md`](../philosophy/working-with-ai.md))
7. **Continuous learning** — concepts over tool churn ([`../learning/2026.md`](../learning/2026.md))

### Tone

- First person ("I") is fine
- Concrete themes over adjectives
- One or two **specific pattern examples** (repository consolidation, API compatibility layer, configurable systems)—not a bullet list of every story
- **No fabricated metrics**
- **No motivational platitudes** ("I love turning ideas into reality")

### Anchor sentence (theme, not mandatory copy)

> I improve software by making it easier to understand, maintain, and evolve.

Support with 2–3 sentences on how that shows up professionally (~10 years, React at scale, full-stack history, current AI-native practice).

### What About should not be

- A pasted résumé summary with every employer listed
- A skills dump
- ML/model training claims
- "Frontend expert" as primary identity

**Draft from:** [`../context.md`](../context.md) · [`../resume/frontend.md`](../resume/frontend.md) or [`../resume/ai-engineer.md`](../resume/ai-engineer.md) depending on campaign—not [`../resume/master-resume.md`](../resume/master-resume.md) verbatim.

## Recommendations

### How to use recommendations effectively

LinkedIn recommendations are **social proof** for themes already documented in the career knowledge base—not a second résumé.

**Themes existing recommendations and future requests should reinforce:**

| Theme | What recommenders should describe with examples |
|-------|--------------------------------------------------|
| **Ownership** | Staying responsible after ship; go-to person on owned systems |
| **Collaboration** | Product, Design, Backend, cross-team work |
| **Technical quality** | Maintainability, review standards, simplification |
| **Product thinking** | Questioning tickets; improving outcomes |
| **Mentoring** | Pairing, explained reviews, growing juniors |

**When requesting a recommendation:**

1. Provide **one specific story** from [`../stories/`](../stories/) or [`../experience/`](../experience/)
2. Ask for **concrete behavior**, not generic praise
3. Vary themes across recommenders—avoid five copies of "great team player"
4. Prefer managers, tech leads, PMs, and senior peers who saw **ownership or architecture** work

**Do not ask recommenders to claim:** formal management title, invented metrics, ML/AI expertise, or scope you did not have (e.g., sole architect of enterprise platform).

→ Request framing: [`../strengths.md`](../strengths.md) · [`../leadership.md`](../leadership.md)

**Note:** Individual recommendation text is **not** stored in this repository—update this section when new themes appear on the live profile.

## Activity

### Posting strategy

**Goal:** Consistency and credibility—not high frequency or influencer performance.

**Recommended topics (when posting):**

- Architecture decisions and trade-offs (simplification, migration patterns)
- AI-assisted engineering **with review discipline**—not hype ([`../philosophy/working-with-ai.md`](../philosophy/working-with-ai.md))
- Lessons learned from personal projects (Free Slot, portfolio, career repo)
- Maintainability, refactoring, and deleting code
- Frontend **and** backend engineering—balanced identity
- Project updates when something **ships to production**
- Technical write-ups when published (link from Featured)

**Cadence:** Occasional is fine—monthly or when something genuinely worth sharing ships. Empty activity is acceptable; low-quality posting is not.

**Avoid:** Engagement bait, generic motivation, hot takes without engineering substance, claiming production RAG/agents before documented.

## What To Avoid

| Avoid | Why |
|-------|-----|
| Buzzword-heavy summaries | Undermines credibility with senior hiring managers |
| Long technology lists in headline or About | Reads as ATS spam; obscures narrative |
| Copying résumé verbatim | Wastes LinkedIn's narrative space; duplicates without search benefit |
| Generic motivational statements | "Passionate about innovative solutions" says nothing |
| Inflated claims | Team size, titles, architecture ownership beyond documented scope |
| Fabricated metrics | No percentages, latency numbers, or revenue unless sourced |
| ML / AI research positioning | Not documented—AI Software Engineer = workflow + production engineering |
| "React developer trying to break into…" framing | Weak vs documented senior narrative |
| Emoji in headline or About | Professional tone per this strategy |
| Contradicting Career Knowledge Base | One source of truth across LinkedIn, résumé, portfolio |

→ Accuracy rules: [`../context.md`](../context.md#instructions-for-ai-assistants)
